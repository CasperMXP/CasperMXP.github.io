Maven简介

Apache Maven is a software project management and comprehension tool.
Based on the concept of a project object model (POM),
Maven can manage a project’s build, reporting and documentation from a central piece of information.

Download Maven

System Requirements

  • JDK 1.7 or above

Installing Apache Maven

  • 确保配置JAVA_HOME环境变量
  • 解压Maven安装文件，添加bin目录到系统环境PATH，以后就可以到处执行mvn命令（最优秀的是%MAVEN_HOME%\bin）
  • 检查安装mvn -v

Configuring Apache Maven

  • MAVEN_OPTS环境变量，设置JVM的内存大小

编写POM.xml文件

创建一个test的文件夹,创建pom.xml文件，并粘贴如下显示内容

  • project元素是pom.xml的根元素，其中声明命名空间以及xsd元素
  • modelVersion元素只能是4.0.0
  • groupId元素定义项目属于哪个组，一般为域名的倒写

  • url元素一般是该项目发布网站地址

编写主代码

  • 默认情况下,Maven约定的项目代码位于src/main/java,所以首先建立目录
  • 默认情况下，创建的Java类的包名应该是groupId.artifactId(com.mycompany.app.myapp)

编写测试代码

  • 默认情况下，Maven约定的测试代码位于src/test/java
  • 默认情况下，创建的包名与要测试的Java代码类的包名一致

Maven常见的约定目录

  • src/main/java存放Java代码
  • src/test/java存放测试代码
  • src/main/resources存放项目资源文件，一般为配置文件(spring,hibernate等)
  • src/test/resources存放项目测试资源文件

坐标(Coordinate)

怎么样引用第三方的Jar包或war包，相对于传统的复制粘贴到lib目录下，在使用maven的时候可以使用坐标来引入第三方的jar或war

坐标的基本元素

  • groupId 定义当前Maven项目隶属的实际项目，一般为域名的倒写
  • artifactId 定义当前实际项目中的一个Maven项目(模块)
  • version 定义Maven项目所处的当前版本
  • packaging 定义Maven项目的打包方式，当不定义的时候，默认值是jar

依赖的基本元素

依赖可以简单理解为就是我们项目开发中要引用的第三方的jar或war包，下面代码就是怎么引入一个junit的依赖

  • dependency 定义一个基本的依赖(三方jar或war包)
  • groupId，artifactId，version为坐标的基本依赖
  • type 依赖的类型，对应坐标的packaging元素，不定义的时候，该元素的默认值是jar,切记依赖是jar or war
  • scope 依赖的范围（见下一小节）

依赖的范围

简单的理解依赖范围，就是我们开发的项目的时候，对于的我们的功能代码，用到spring-core.xx.jar的类，如果不引用该jar包，编译的
时候就会出错，这种就是常说的编译的classpath,还有针对的功能的测试代码，只是作为测试使用，针对JUNIT的jar包就是属于测试classpath,

  • compile编译依赖范围，如果没有指定，默认会使用该依赖。简答的理解打包的时候会打包到代码里。
  • test测试依赖范围，只是针对测试代码有效，简单的理解就是不会打包到发布的包里，典型的例子就是JUNIT。
  • provided已提供的依赖范围，对编译和测试有效，但是对于运行时无效，典型的例子就是servelt-api，运行时容器已经提供。
  • runtime运行时依赖范围，对测试和运行时有效，对编译无效，典型的是JDBC驱动，只有在运行和测试的时候才需要JDBC驱动的实现类。

传递性依赖

简单的理解就是，一个A项目依赖spring-core.xxx.jar，而spring-core.jar又依赖commons-logging.jar,此时相对于A项目就传递依赖
commons-logging.jar

依赖范围影响传递性依赖

依赖范围 compile test provided runtime
compile compile - - runtime
test test - - test
provided provided - provided provided
runtime runtime - - runtime

依赖调解

  • 路径最近者优先，A->B->C->D(1.0), A->E->D(2.0),最终D(2.0)会被解析使用。
  • 第一声明优先，路径相同情况下，在POM.xml中定义的先后顺序在前面的会被解析使用。

排除依赖

在很多的时候，传递性依赖会隐式引入很多依赖，这样极大的简化了项目管理，但是有些时候会带来其他的影响。
排除依赖主要就是排除传递性依赖在依赖定义的时候使用exclusions元素，其中可以包含多个exclusion元素

生命周期和插件

三套生命周期

maven的生命周期就是为了对所有的构建过程进行抽象和统一。这个生命周期包含了项目清理，初始化，编译，测试，打包，集成测试，验证，
部署和站点生成等几乎所有构建的步骤。

clean生命周期

  • pre-clean 执行清理前需要执行的一些工作
  • clean 清理上一次构建产生的文件
  • post-clean 执行清理后需要执行的一些工作

default生命周期

  • validate
  • initialize
  • generate-sources
  • process-sources 处理项目主资源文件，一般来说，是对src/main/resources目录内容，复制到项目输出的主classpath目录中。
  • generate-resources
  • process-resources
  • compile 编译项目主代码，一般来说就是编译src/main/java到主classpath目录中
  • process-classes
  • generate-test-sources
  • process-test-sources 处理项目测试资源文件
  • generate-test-resources
  • process-test-resources
  • test-compile
  • process-test-classes
  • test 使用单元测试框架进行测试
  • prepare-package
  • package 打包
  • pre-integration-test
  • integration-test
  • post-integration-test
  • verify
  • install 将包安装到本地仓库
  • deploy 将包复制到远程仓库

maven内置插件绑定

maven为了简化使用，在很多核心的生命周期已经内置绑定了一些插件

生命周期阶段 插件目标 执行任务
pre-clean,clean,post-clean maven-clean-plugin:clean 清理项目
pre-site,site,post-site maven-site-plugin:site 生成站点
site-deploy maven-site-plugin:deploy 部署站点
process-resources maven-resources-plugin:resources 复制主资源文件到主输出目录
compile maven-compile-plugin:compile 编译主代码到主输出目录

test maven-surefire-plugin:test 执行测试用例
package maven-jar-plugin:jar 创建项目jar包
install maven-install-plugin:install 将项目输出构件安装到本地仓库
deploy maven-deploy-plugin:deploy 将项目输出构建部署远程仓库
