基本概念

React只是DOM的一个抽象层，并不是Web应用的完整的解决方案。有2个方面，没有涉及：

  • 代码结构
  • 组件之间的通信

Redux的使用场景

  • 用户的使用方式复杂
  • 不同的身份的用户有不同的使用方式
  • 多个用户之间可以协作
  • 与服务器之前大量交互，或者使用websocket
  • View要从多个地方获取数据源
  • 多交互，多数据源

从组件的角度看，如果应用有以下场景，可以考虑使用Redux

  • 某个组件的状态需要共享
  • 某个状态需要任何地方都可以拿到
  • 一个组件需要改变全局状态
  • 一个组件需要更改另一个组件的状态

设计思想

  • web应用是一个状态机，视图与状态一一对应
  • 所以的状态，保存在一个对象里

基本概念和API

Store

Store 就是保存数据的地方，你可以把它看成一个容器。整个应用只能有一个 Store。

Redux 提供createStore这个函数，用来生成 Store。

State

Store包含所有的数据。如果想得到某个时点的数据，就要对Store生成快照，这种数据就叫做State。

Redux 规定， 一个 State 对应一个 View。只要 State 相同，View 就相同。

Action

State的变化会导致View的变化，但是用户接触不到State,只能接触到View.Action就是View发出的通知，表示State发生变化了
Action是一个对象，其中type属性是必须的，表示Action的名称。其他属性可以自由设置

Reducer

store收到Action后，必须给出一个新的state,这样view才会发生变化，这个state的计算过程就是reducer.
reducer是一个函数，接受当前action和当前state作为参数，返回一个新的state

store.subscribe()

Store 允许使用store.subscribe方法设置监听函数，一旦 State 发生变化，就自动执行这个函数。

React-Redux

React-Redux将所有组件分成2大类，UI组件和容器组件

UI组件的特征

  • 只负责UI的呈现，不带有任何的业务
  • 没有状态即不使用this.state
  • 所有数据都由参数this.props提供
  • 不使用redux的任何API

容器组件

  • 负责管理数据和业务逻辑，不负责UI的呈现
  • 带有内部状态
  • 使用Redux的API

React-Redux 规定，所有的 UI 组件都由用户提供，容器组件则是由 React-Redux 自动生成

connect()

React-Redux 提供connect方法，用于从 UI 组件生成容器组件。connect的意思，就是将这两种组件连起来。

  • mapStateToProps，负责输入逻辑，即将state映射到 UI 组件的参数（props）
  • mapDispatchToProps，输出逻辑，即将用户对 UI 组件的操作映射成 Action

mapStateToProps()

mapStateToProps是一个函数。它的作用就是像它的名字那样，建立一个从（外部的）state对象到（UI 组件的）props对象的映射关系。
